Dependency Diagram

Utilization-Aware Scaling

How It Works

The controller connects to workload clusters via CAPI kubeconfig Secrets (<cluster>-kubeconfig) to observe real node utilization. Both pod resource requests (scheduled utilization) and metrics-server data (actual utilization) are used.

  • Upscale (75% default): Triggers if max(scheduled, actual) exceeds threshold for either CPU or memory. Selects a bigger VM plan with savings threshold relaxed to 0.
  • Downscale (5% default): Triggers only if min(scheduled, actual) is below threshold for both CPU and memory. Additionally, a scheduling simulation must confirm all pods fit on the smaller plan.
  • In range: Falls back to the existing cost-optimization path (DirectionAny).
  • Graceful degradation: If the kubeconfig Secret is missing or metrics-server is unavailable, the controller falls back to static constraints with no behavioral change.

Scheduling Simulation

Before allowing a downscale, the scheduler package simulates bin-packing all non-DaemonSet, non-mirror pods onto virtual nodes with the candidate plan's capacity. For each pod it checks:

  • Resource fit: pod.requests.cpu ≤ allocatable_cpu and pod.requests.memory ≤ allocatable_ram
  • Tolerations: Pod must tolerate all NoSchedule/NoExecute taints on the node
  • NodeSelector: All pod.spec.nodeSelector keys must match node labels
  • NodeAffinity: Required matchExpressions (In, NotIn, Exists, DoesNotExist, Gt, Lt) are evaluated

If any pod cannot be placed, downscale is blocked and the blocking pods are reported in the SchedulingSimulation condition.

Horizontal Scaling

How It Works

Horizontal scaling adjusts the replica count of a MachineDeployment (adding/removing worker nodes) based on workload demand, complementing vertical scaling (which changes VM plan size).

  • Scale up: When unschedulable (Pending) pods are detected in the workload cluster, the scaler increases replicas by 1. Subsequent reconciles add more if pods remain pending.
  • Scale down (multi-phase): When both CPU and memory utilization are below scaleDownThreshold (default 25%), and a bin-packing simulation confirms all non-system pods fit on N-1 nodes, the scaler initiates a cordon → drain → verify → reduce sequence across reconcile cycles.
  • Guards: Respects minReplicas/maxReplicas bounds. Skips scaling when a rollout is in progress (readyReplicas < currentReplicas) or a drain is in progress (status.drainingNode set). Stabilization window prevents flapping.
  • CAPI integration: Patching MachineDeployment.spec.replicas triggers CAPI's machine deletion controller. The scaler handles cordon+drain itself to verify pod rescheduling before reducing replicas.

Horizontal Scale-Down Safety

Before removing a node, the scheduler's bin-packing simulator verifies all workload pods fit on the remaining nodes:

  • Synthetic plan: Node allocatable resources are used to construct a virtual plan (no dependency on VPSie plans API)
  • First-fit-decreasing: Pods sorted by resource requests (largest first), placed onto N-1 virtual nodes
  • Constraint checks: Resource fit, taints/tolerations, nodeSelector, required nodeAffinity
  • Excluded: DaemonSet pods and mirror pods (auto-schedule everywhere)
  • If unsafe: Scale-down is blocked and the reason is logged + set in the HorizontalScaling condition

Multi-Phase Node Drain

Scale-down uses a multi-phase cordon → drain → verify → reduce flow across reconcile cycles to ensure zero pod disruption:

  1. Phase 1 (initial reconcile):
    • Select the least-utilized node (lowest combined CPU+memory request ratio)
    • Cordon the node (mark unschedulable via node.spec.unschedulable=true)
    • Drain the node (evict all non-DaemonSet, non-mirror pods via policyv1.Eviction)
    • Set status.drainingNode and status.drainingStartedAt to track the operation across reconciles
  2. Phase 2 (subsequent reconcile):
    • Timeout check: If drainingStartedAt > 5 minutes ago → uncordon the node, clear drain state, emit DrainTimeout warning event
    • Check GetNonSystemPodCount() on the draining node
    • If pods remain → requeue (keep waiting for eviction to complete)
    • If 0 pods remain → verify no new pending pods appeared (safety check)
    • If pending pods exist → uncordon the node, abort scale-down, clear drain state
    • If all clear → reduce replicas, record event, clear drain state

This ensures pods are fully rescheduled and running on other nodes before the MachineDeployment replica count is decremented. Nodes are always uncordoned on abort/timeout to prevent wasted capacity.

CAPI v1beta2 ReadyReplicas

md.Status.ReadyReplicas (*int32) in CAPI v1beta2 requires ALL machine conditions to be True (including infra conditions like FirewallAttached, PublicIPAssigned). Since CAPV doesn't always set these for workers, this stays at 0.

The scaler uses md.Status.Deprecated.V1Beta1.ReadyReplicas (int32) instead, which counts machines with Ready=True only.

Data Flow: Reconcile Cycle

 1. ScalingPolicy change / MachineDeployment change triggers reconcile
 2. Resolve target MachineDeployment by spec.targetRef
 3. Resolve API key: policy.credentialsRef → fallback VPSieCluster.credentialsRef
 4. Ensure pricing cache is fresh → GET /api/v2/plans/category
                                    → POST /api/v2/resources per category
 5. Read current VPSieMachineTemplate → resolve resourceIdentifier to plan
 6. determineDirection():
    a. Read <cluster>-kubeconfig Secret from management cluster
    b. List Nodes (via CAPI Machines status.nodeRef) → List Pods → Get NodeMetrics
    c. Calculate scheduled utilization: sum(pod.requests) / allocatable
    d. Calculate actual utilization: metrics-server usage / allocatable (if available)
    e. Evaluate thresholds:
       • Upscale: max(sched, actual) > 75% for CPU or memory → DirectionUp
       • Downscale: min(sched, actual) < 5% for both → run scheduling simulation
         • Simulation safe → DirectionDown
         • Simulation blocked → DirectionAny (fallback)
       • In range → DirectionAny
    f. Update status.currentUtilization + Prometheus metrics
 7. reconcileHorizontal() (if spec.horizontal.enabled):
    a. If status.drainingNode set → verify drain complete:
       • drainingStartedAt > 5min? → timeout: uncordon node, clear drain state
       • Pods remaining? → requeue (wait for eviction)
       • 0 pods + no pending → reduce replicas, clear drainingNode
       • 0 pods + pending exist → abort: uncordon node, clear drainingNode
    b. List pending pods in workload cluster
    c. Pending pods? → scale up by 1 (up to maxReplicas)
    d. No pending + CPU & memory < scaleDownThreshold?
       → Run bin-packing simulation on N-1 nodes
       → Unsafe: block, log blocking pods
       → Safe: cordon least-utilized node → drain (evict pods) → set status.drainingNode
    e. Stabilization window + rollout-in-progress + drain-in-progress guards
 8. Run selector.Select() with direction: filter → directional filter → fit → score → threshold
 9. If dry-run: log recommendation, set DryRun phase, done
   If switch: create new VPSieMachineTemplate → patch MD infrastructureRef
              → CAPI handles rolling update (cordon, drain, replace)
10. Update status: currentPlan, recommendedPlan, savings, utilization, conditions, phase
11. Requeue after 60s

Safety Properties

  • No behavioral change when WorkloadClients is nil: Without workload access, falls back to DirectionAny with static MinCPU/MinRAM constraints (original behavior).
  • Conservative downscaling: Requires BOTH CPU and memory below 5% using min(scheduled, actual), AND scheduling simulation confirms all pods fit on the smaller plan.
  • Eager upscaling: Triggers if EITHER CPU or memory exceeds 75% using max(scheduled, actual). Savings threshold is relaxed to 0 (capacity > cost).
  • Graceful degradation: Kubeconfig secret missing? Metrics-server down? Calculator error? → All degrade gracefully to DirectionAny with conditions set for observability.
  • DaemonSet & mirror pod exclusion: Scheduling simulation ignores pods that auto-schedule everywhere, preventing false positives.
  • Horizontal bin-packing: Before removing a node, bin-packing simulation verifies all pods fit on N-1 nodes. Scale-down blocked if pods would become unschedulable.
  • Multi-phase drain: Scale-down cordons and drains the least-utilized node, then waits for all pods to be rescheduled before reducing replicas. If new pending pods appear post-drain, scale-down is aborted and node is uncordoned. Tracked via status.drainingNode + status.drainingStartedAt across reconcile cycles.
  • Drain timeout: 5-minute timeout prevents stuck drains (e.g. PDB-blocked pods). On timeout, the node is automatically uncordoned and drain state is cleared. Emits a DrainTimeout warning event.
  • Uncordon on abort: Whenever a drain is aborted (pending pods or timeout), the node is uncordoned to prevent wasted capacity from a cordoned-but-not-removed node.
  • Scale 1 at a time: Both horizontal scale-up and scale-down change replicas by 1 per reconcile, preventing over-/under-provisioning oscillation.
  • Rollout awareness: Horizontal scaling skips when readyReplicas < currentReplicas (machines still provisioning) or drain is in progress (status.drainingNode set). Uses CAPI v1beta1 deprecated field for accurate ready counts. Stalled rollouts (>15min) emit warning events.
  • Observability: 7 condition types, 29 reason codes, 9 Prometheus metrics provide full visibility into every decision.
